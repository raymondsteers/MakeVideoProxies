About:

This program, like many others, is mostly a very complicated wrapper around FFMPEG which does all the heavy lifting. If you have questions about FFMPEG then google your life away on the subject.

I wrote this program over the space of a month while I was in the jungle in Columbia South America acting in a movie. I had lots of time on my hands in the hotel and had only brought my old slow MacBookPro which was far too slow to edit some of the HD footage I was shooting on my own as a way to both kill time and help explore my character's backstory.

The initial program was just a static Perl script that I would run on the command line to batch convert all the 1920x1080p and 1920x810p stuff down to 640x360.

I make no claims that this program is perfect or even very good. If it helps you then great. If it does not then well you get what you pay for.

I will be changing the name of this program to 'Tedium' at the next version. Because editing is tedious and this was always thought of as a tool to do tedius stuff automatically.

Happy converting...

-Raymond Steers

Output FPS Help:

Leaving this box empty means that each output file will be created with the same frame rate as it's source file.

Putting a number in this box like 24 or 23.976 will force ALL files to be converted to that framerate.

The duration of the output video will remain the same no matter what the framerate change is (hopefully).

Only Go One Input Level Deep Help:

UN-CHECKING this box means that the program will dig down infinite levels into the sub directories (folders) of the chosen input directory (folder) and find ALL video files.

It is a BAD IDEA to uncheck this IF you set your output directory as a sub-directory of the input directory, as when you run it a second time you will eat a hole in the fabric of spacetime. You have been warned.

Because of the danger of this, we will ALWAYS reset this box to CHECKED and not save it's UN-CHECKED state.

Append Files to Queue Help:

Checking this will add all the files you find each time you click the 'SCAN' button to the processing queue. NOT checking this box (default) will empty the queue before each click of the 'SCAN' button.

If for example, you are using this program nightly during a multi-day shoot and want to copy new files off your camera cards into the same source folder, having this box checked makes sure you don't re-convert files from previous days and waste time. Auto Crop Help:

Cropping is done BEFORE scaling.

First we compare the output aspect ratio to the input aspect ratio if they are the same then there is nothing to do cropwise.

I think anything less than a 1.5% aspect ratio change is unnoticable and not worth fucking around with, in which case we ignore the cropping that would be asked for by the autocrop feature on.

Then we need to apply the aspect ratio of the output to the input in both directions W or H as more important.

If the output aspect ratio is greater than the input aspect ratio then we need to crop top and bottom (crop height).

If the output aspect ratio is less than the input aspect ratio then we need to crop left and right (crop width).

We do NOT pad with black in either direction as that's really just a waste of data and all video players can do that on the fly in full screen anyway.

Obviously cropping is done symmetrically.

Create HTML Gallery Help:

Checking this box will create a file in the output directory called 'gallery.html' which you can then just double click on and open it in a browser. This will allow you to see a usable gallery of all the proxies made and play them. It will be overwritten on each run.

If in the advanced section you put anything in the 'HTML Gallery File Name' box THEN that will be used as the file name. It will NOT allow you to use any characters besides 0-9, a-z, A-Z, and underscore. All other characters will be removed. Also you do not need to add '.html' at the end and if you do that will be removed also and then added later when created.

If you change the 'File Name Variables' dropdown to 'DateTime Stamp' THEN the gallery file name will be either 'gallery-YYYY-MM-DD-hh-mm-ss.html' OR it will be whatever you typed in the file name box plus '-YYYY-MM-DD-hh-mm-ss.html'

If you change the 'File Name Variables' dropdown to 'Random' THEN the gallery file name will be either 'gallery-3d0b3e91-8181-41e7-a3cc-c4870d4bd64c.html' (a RFC4122 version 4 UUID) OR it will be whatever you typed in the file name box plus a UUID plus '.html'

In either case, adding a variable will keep the old gallery html file from getting overwritten. This is part of the intent obviously.

As to WHY you might want to do this:

IF you want to keep a history of what the output state of videos are then datetime stamps in the name will do that providing you don't go deleting or overwriting the video files.

IF you want to post the gallery to the web somewhere so that you can share it will others, then using random gallery file names provides you with some security through obsecurity as the file name will not be guessable or tumbleable (at least not easily enough to be finished before your film has been over for a few years)

Video Codec Help:

Well....what to say here? The rabbit hole of video codecs might start here: https://ffmpeg.org/ffmpeg-codecs.html (paste that in your browser and have fun reading)

What's interesting is that the definition of "codec" is: "Codecs are compression technologies and have two components, an encoder to compress the files, and a decoder to decompress."

However... that's kind of bullshit. "Codecs" sadly do NOT compress data. They throw out data they believe you won't notice or care about as much. Now, something like ZIP, actually compresses data, because when you uncompress it the resulting data is identical to the original. Video and Audio Codecs with few exceptions do nothing of the kind. A MUCH MUCH better term would be perhaps "Decimator" and if we used proper nomenclature in the media then we would not be shocked when video looks like total garbage when "over-compressed" because we would call it "over-decimated".

Pixel Format Help:

Videos, images and other visual media contain a value called Pixel Format. It describes the layout of each and every pixel in the image data of a picture or video. The Pixel Format is related to the color channel or color space of the visual data. It describes how the color data is encoded and organized. Usually, the value given for the Pixel Format in a file includes the bits per pixel (bpp) and the color channel or model. This includes, among others: YCbCr RGB YUVJ

What you need to know here is that on this drop down list the pixel formats shown also have in parenthesis next to them a list of what video codesc they are compatabile with. So... DON'T choose a pixel format that is incompatible with the video codec you have chosen.

h264/h265 Preset Help:

A preset is a collection of options that will provide a certain encoding speed to compression ratio. A slower preset will provide better compression (compression is quality per filesize). This means that, for example, if you target a certain file size or constant bit rate, you will achieve better quality with a slower preset. Similarly, for constant quality encoding, you will simply save bitrate by choosing a slower preset.

Use the slowest preset that you have patience for. The available presets in descending order of speed are:

  • ultrafast
  • superfast
  • veryfast
  • faster
  • fast
  • medium
  • slow
  • slower
  • veryslow

h264/h265 CRF Help:

The range of the CRF scale is 0â€“51, where 0 is lossless, 23 is the default, and 51 is worst quality possible. A lower value generally leads to higher quality, and a subjectively sane range is 17â€“28. Consider 17 or 18 to be visually lossless or nearly so; it should look the same or nearly the same as the input but it isn't technically lossless.

The range is exponential, so increasing the CRF value +6 results in roughly half the bitrate / file size, while -6 leads to roughly twice the bitrate.

Choose the highest CRF value that still provides an acceptable quality. If the output looks good, then try a higher value. If it looks bad, choose a lower value.

Note: The 0â€“51 CRF quantizer scale mentioned on this page only applies to 8-bit x264. When compiled with 10-bit support, x264's quantizer scale is 0â€“63. You can see what you are using by referring to the ffmpeg console output during encoding (yuv420p or similar for 8-bit, and yuv420p10le or similar for 10-bit). 8-bit is more common among distributors.

Prores QScale Help:

This parameter determines the quality of the resulting prores movie - both the resulting size and bitrate. 0 means best and it goes up to 32 which is worst. From empirical testing we've found that a qscale of 9 - 13 gives a good result without exploding the space needed too much. 11 would be a good bet, 9 if a slightly better quality is required. When space is not a problem, go with qscale 5 or less, but approaching zero the resulting clip will be extremely large and the bitrate will be so high that it will stop being playable on normal equipment. The "vendor" argument when set to "apl0" tricks quicktime and Final Cut Pro into thinking that the movie was generated on using a quicktime prores encoder.

Audio Codec Help:

Go here and read this: https://trac.ffmpeg.org/wiki/Encode/HighQualityAudio

If you have for example files from a cinema grade video camera then it is possible that it might contain multitrack uncompressed source audio and you might very want to reduce the video rezolution but keep all the audio tracks intact. In that case you would want to choose 'copy' for the codec and set the bitrate and channels to 'default', failing to do this you will end up mixing down your tracks to either stereo or mono.

Conversely, if you are making a bunch of small preview files that you want to play fast over teh web or on mobile then you might want to choose AAC or MP3 and set the channels to 'mono' and then reduce the bitrate to something low like '32k' which will give you crappy audio but it will greatly reduce the file size. Understand the default bitrate for AAC is 128k/sec, so choosing 'default' and choosing AAC or MP3 will give you 128k.

Burn In Help:

You have 2 items that can be burned into the output video. They can be used individually or together. Putting text into the 'Burn In Text' field will activate that feature. Checking the 'Burn In Timecode' box will activate that feature.

Keep in mind that you should make a test with a single short video to make sure that the result looks good before you unleash it on a bunch of videos and waste a lot of time creating bad results.

You can set the location, color, size, and border, of each item.

When setting the color, you will find a color picker will pop up when you click on the little color squares and you can not only change the color but also the opacity (alpha). This is helpful if you want to make watermarks.

The system makes no attempt to stop you from putting more text in the box than will show up on the screen. In other words your text can easily run off the screen if you have too many characters and too big a font size.

The text input control for the burn in text is ALSO a preview of how it will look in the final video. Consider the purple-blue gradient background to be your video. As you change the resolution of the output this box will size itself to those same dimentions. If you are trying to scale to 1920 wide and your computer screen is only 1280 wide then you can hit one of the 4 zoom buttons to shrink the preview. Also clicking the Show Advanced button on and off will automatically zoom it down to a usable size for you.

This preview is NOT 100% accurate!!! That would be rather impossible actually unless we had FFMPEG make a single jpg frame of the video and then loaded it each and every time you changed a setting which would be super slow. The text border preview is not perfect (which you will be warned about the first time you crank it up more than zero), and setting text all the way as wide as possible to the edge of the video is not advisable as FFMPEG might scale the font very slightly differently.

NOT ALL your fonts will work right. Especially if you are on Windows and you are using extended characters above the standard Latin set. For example if you try to type Russian using a font that has no Russian characters then you will get weird boxy garbage stuff. This is a Windows problem and not a problem with this program. Mac fonts generally don't have this issue as Mac (having a UNIX core) knows how to actually do UTF8 correctly (more or less). The point is if you choose a font and see no change in the text, but most of your other fonts do change the text, please don't report this as a bug. It's not a bug, it's you baby.

You CAN insert METADATA VARIABLES into the text. For example if your camera inserts metadata into it's video files with interesting and helpful information like ARRI camera will insert the metadata tag "com.arri.camera.ExposureIndexAsa" which means ISO actually, and you want this burnt in, then you can use that as a variable in the burn in text box by putting a tilde character before and after it like so: "~com.arri.camera.ExposureIndexAsa~" and this will get replaced with the value. You can use as many metadata tags as you want and you can mix them with other normal text. If the metadata tag is not found it will be simply removed from the output. You can find your metadata tags by clicking the little grey "details" buttons in the activity log that are created for each video file after scanning. They will be in the first section indented under "format" and then under "tags".

The Timecode does NOT preview. Also it does not have a changeable font. It will use whatever the default font on your system is. If you really want to know what it will look like then just set the text to Arial and type '00:00:00.000' into the text preview and change the settings and that should give you an idea. Note, if you set the screen location of the text AND the timecode to be the same then you will have ugly crap (duh).

As you can see one file uses 'modification_date' and one file uses 'creation_time' and I have found other files that use 'CREATION_TIME' and then many others that have NO creation datetimestamp at all.

So we search for a datetimestamp of some kind and then do some ugly calc to create the right offset for the best_effort_timestamp_time in each frame. THAT is your timecode. So, IF your file has no datetimestamp, THEN you will NOT see a date to the left of the timecode and the timecode will always start at zero. IF there is a datetimestamp then as long as the camera had the correct time set then you should see the actual date and time that each frame was captured.

HOWEVER... you will be seeing the local time. So if your camera is in one time zone and they start shooting at 3am then you will see timecode that says 3am, no matter if you are in a different time zone or not.

The 'Border' fields are set to zero by default and they mean pixels of thickness of how much outline (stroke) the text gets. Leaving them at 0 makes no stroke effect.
